# Agent 运行时设计

## 设计目标

Agent 运行时负责把用户问题转换为可执行计划，按需调用知识库与外部信息工具，再基于工具结果生成带引用的答案。它不包含任何特定行业、文档名称或知识库关键词规则，新增能力通过工具注册完成。

核心约束：

- 规划、执行和答案生成职责分离。
- 工具由统一注册表管理，运行时只依赖工具契约。
- 运行时按“规划、执行、观察”循环推进，资料充分后进入答案生成。
- 同一轮内没有依赖关系的工具可以并行执行。
- 所有工具都有参数 Schema、超时、长度和数量限制。
- 工具结果属于不可信输入，只能作为回答依据，不能覆盖系统约束。
- 执行步骤通过 SSE 实时返回，并随消息持久化。

## 执行模型

```text
用户问题
   |
   v
计划生成 <--- 可用工具清单 / 历史调用 / 当前观察
   |
   v
计划校验 ---- 工具白名单 / 参数长度 / 去重 / 最大调用数
   |
   v
工具执行 ---- 并行调度 / 独立超时 / 请求取消
   |
   v
结果观察 ---- 片段去重 / 外链去重 / 观测内容裁剪
   |
   +---- 资料不足且仍有预算：返回计划生成
   |
   v
答案生成 ---- 仅基于工具结果 / 引用过滤 / 流式输出
   |
   v
消息落库 ---- 答案 / 引用 / 指标 / Agent 步骤
```

状态机仅允许 `initialized -> planning -> executing -> observing -> planning/synthesizing -> completed/failed` 中定义的转移。最大迭代数、单轮工具数、总工具数、跨轮重复调用和无有效结果共同构成终止条件，避免模型形成无限工具循环。

首次规划不可用或返回非法结构时，运行时退化为通用知识检索计划；后续规划失败时直接使用已有观察生成回答。退化策略只依赖已注册工具，不通过问题关键词判断业务意图。

## 工具契约

每个工具必须提供：

| 能力 | 要求 |
| --- | --- |
| 名称 | 全局唯一、稳定、使用小写下划线 |
| 描述 | 说明适用场景，不包含特定知识库内容 |
| 参数 | 使用 JSON Schema 声明，拒绝未知字段 |
| 执行 | 接受请求上下文，正确响应超时和取消 |
| 输出 | 返回模型可读结果，以及结构化引用或指标 |
| 错误 | 日志记录内部原因，对模型和前端返回安全描述 |

当前内置工具：

| 工具 | 职责 |
| --- | --- |
| `knowledge_search` | 混合检索、重排与引用裁剪 |
| `knowledge_overview` | 汇总知识库文档与主题分布 |
| `document_navigation` | 定位文档、章节与可阅读内容 |
| `web_search` | 在用户启用联网搜索时补充公开网页资料 |

新增工具只需实现工具契约并注册到注册表。Chat Logic、Handler 和 Agent 调度器不应因新增工具而增加分支。

## 安全与可靠性

- 总执行时间和单工具时间分别受限。
- 问题、工具参数、步骤详情和观测结果均限制最大长度。
- 工具调用只允许来自配置白名单，非法或重复调用会被丢弃。
- 跨迭代重复的工具与参数组合不会再次执行。
- 关闭联网搜索时，规划器看不到联网工具，运行时也不会执行它。
- SSE 写入失败会取消并行工具，避免客户端断开后继续消耗资源。
- 最终只保留答案实际引用的知识片段和外部链接。
- 生成失败、超时和工具错误写入日志，但不把内部错误或密钥暴露给模型与用户。

## 性能策略

- 按需迭代：每轮只在当前观察不足时继续规划，达到上限后强制进入答案生成。
- 并行执行：没有依赖关系的工具并行运行，结果按原计划顺序汇总。
- 复用服务：检索、概览和导航工具调用现有领域服务，不复制数据库与向量检索逻辑。
- 有界上下文：工具数量、候选片段、引用数量和观测文本均有上限。
- 连接复用：默认对话客户端、向量化客户端和其他基础服务由应用上下文统一创建并复用；切换对话模型时通过 Provider 工厂创建对应客户端。
- 可测量：`/metrics` 暴露 HTTP、状态转移、Agent、工具、模型、检索和 Worker 指标；回答内保留 Recall@K、候选数和最终引用数。

指标标签只使用路由、阶段、工具、任务类型和结果等有限集合，不包含用户 ID、知识库 ID、问题、文件名或模型 ID，避免时序数量失控。性能参数应通过真实知识库测试调整，重点观察端到端延迟、状态机迭代数、检索 Recall@K、工具超时率、生成失败率和最终引用相关度。

## 扩展边界

适合扩展为工具的能力包括结构化数据库查询、内部业务 API、计算器、代码执行沙箱和更多搜索服务。需要人工审批或长时间运行的任务，应在现有运行时之上增加可持久化工作流层，而不是让单次聊天请求无限执行。

当前实现不引入第二种编程语言。Go 已能覆盖并发调度、流式传输、服务复用和部署需求，额外运行时会增加构建、部署和监控成本。只有在 OCR、模型推理或数据科学组件存在成熟且不可替代的生态依赖时，才应把独立能力作为边界清晰的服务引入。
