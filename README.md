Enterprise-RAG 将分散的制度、规范、手册和技术资料整理为统一知识库。用户选择知识库并提问后，系统从原文中检索依据、生成回答并返回引用，形成从文档导入到可信问答的完整闭环。

## 项目特点

| 结构化入库 | 精准检索 | 可信回答 | 可量化优化 |
| :--- | :--- | :--- | :--- |
| 识别标题、段落、页码和表格，尽量保留文档上下文 | 根据问题意图组合改写、语义召回、关键词召回与重排 | 只展示真正支撑回答的内部文档或外部网页来源 | 使用 Recall@K、路由准确率和耗时评估检索效果 |

项目不把 RAG 简化为“向量检索后调用模型”。文档结构、问题类型、候选质量、引用有效性和失败恢复都属于完整问答链路的一部分。

## 系统架构

![Enterprise-RAG 系统架构](./docs/system-architecture.svg)

文档处理与在线问答独立运行：前者通过 NATS 和 Worker 异步建立索引，后者通过问题路由和检索服务实时组织上下文。PostgreSQL、MinIO 与 Milvus 分别保存业务数据、原始文件和向量索引，删除与重建时保持数据一致。

## 核心链路

| 文档入库 | 在线问答 |
| --- | --- |
| 多文件批量上传 | 选择知识库和对话模型 |
| Markdown、TXT、PDF、DOC、DOCX、PPTX 结构化解析 | 识别事实问答、知识概览、文档导航和复合问题 |
| 解析、切块、向量化并发处理 | Query Rewrite、语义检索与关键词召回 |
| NATS 任务流转与 Worker 幂等消费 | 候选重排、相似度过滤与引用裁剪 |
| 失败定位、任务重试和状态追踪 | SSE 流式回答、联网补充与来源定位 |
| 删除原文件、业务记录和向量索引 | 资料不足时明确回答无法确定 |

```text
文件  →  解析  →  切块  →  向量化  →  建立索引
问题  →  路由  →  混合召回  →  重排  →  回答 + 引用
```

## 快速开始

### 环境要求

| Docker | Go | Node.js | pnpm |
| --- | --- | --- | --- |
| Docker Compose | `1.26+` | `20.19+` 或 `22.12+` | `9+` |

### 启动基础服务

```bash
docker compose up -d
```

基础服务包括 PostgreSQL、NATS、MinIO、Etcd 和 Milvus。

### 启动后端

```bash
export OPENROUTER_API_KEY=你的密钥
cd backend
go run .
```

后端默认地址：`http://localhost:9999`

### 启动前端

```bash
cd frontend
pnpm install
pnpm dev
```

前端默认地址：`http://localhost:5173`

## 使用流程

1. 注册并登录系统。
2. 创建知识库并上传文档。
3. 等待文档状态变为“已索引”。
4. 进入助手页面，选择知识库和模型后提问。
5. 阅读回答，并通过引用来源核查原文。
6. 在日志页面查看处理状态；在检索评估中验证召回效果。

## 质量边界

- 扫描图片、复杂图表、加密文件和低质量文档可能影响解析精度。
- 联网资料只作为知识库之外的补充，不能替代企业内部正式依据。
- 固定评估集用于比较检索配置，不代表模型回答在所有问题上都正确。
- 法律、医疗、财务和生产控制等高风险结论仍需专业人员确认。
